# Markdown 작성 기준

기술자료의 기준 원고는 Markdown이다. HTML, PDF, ePub, PPT/PPTX 등은 Markdown 기준 원고에서 변환되는 출판 산출물로 취급한다.

## 1. 문서 구조

- 하나의 문서는 하나의 `#` 제목만 가진다.
- 제목 계층은 `#`, `##`, `###`, `####` 순서로 사용하고 건너뛰지 않는다.
- 제목에는 번호를 붙일 수 있지만, 같은 문서 안에서 일관되게 사용한다.
- 내부 작업 메모는 최종 산출물에 남기지 않는다.
- 최종 산출물에는 `TODO`, `검증 필요`, `출처 필요`를 남기지 않는다.

## 2. 문단과 목록

- 한 문단에는 하나의 주제를 담는다.
- 긴 설명은 문단, 표, 목록 중 가장 읽기 쉬운 형태로 나눈다.
- 중첩 목록은 필요할 때만 사용한다.
- 절차는 순서 있는 목록을 사용한다.
- 병렬적인 항목은 순서 없는 목록을 사용한다.

## 3. 코드 블록

- 코드 블록에는 언어 식별자를 붙인다.

````md
```python
print("hello")
```
````

- 실행이 중요한 코드는 입력, 출력, 실행 조건을 함께 설명한다.
- 코드가 일부만 발췌된 경우 생략한 맥락을 설명한다.
- 최종 산출물의 코드는 가능한 한 실행 검증한다.

## 4. 수식

- 수식은 Markdown 변환 도구가 처리할 수 있는 LaTeX 표기법을 사용한다.
- 인라인 수식은 `$...$`를 사용한다.
- 블록 수식은 `$$...$$`를 사용한다.
- 수식에는 기호, 단위, 가정, 적용 조건을 함께 설명한다.
- 같은 기호는 문서 전체에서 같은 의미로 사용한다.

## 5. 그림과 표

- 그림은 상대 경로로 참조한다.
- 그림에는 설명 가능한 캡션을 붙인다.
- 표는 너무 넓게 만들지 않는다.
- 넓은 표는 여러 표로 나누거나 목록으로 바꾼다.
- 그림과 표가 핵심 설명을 대신하게 하지 말고, 본문에서 해석을 제공한다.

## 6. 링크와 참고문헌

- 내부 링크는 Markdown 앵커 또는 상대 경로를 사용한다.
- MkDocs 내부 페이지 링크는 생성된 HTML 경로가 아니라 대상 `.md` 파일을 가리킨다.
- 챕터 파일을 이동하거나 이름을 바꾸면 해당 파일을 참조하는 상대 링크도 함께 수정한다.
- 외부 링크는 안정적인 공식 출처를 우선한다.
- 논문, 표준, 공식 문서는 가능하면 제목, 저자/기관, 연도, 링크를 함께 기록한다.
- 참고문헌 형식은 한 문서 안에서 일관되게 유지한다.

## 7. 기술 표기

- 같은 개념에는 같은 용어를 사용한다.
- 처음 등장하는 약어는 풀어 쓴 뒤 약어를 병기한다.
- 버전 의존 정보는 버전 또는 확인 날짜를 표시한다.
- 명령어, 파일명, 함수명, 변수명은 인라인 코드로 표시한다.

## 8. 변환 친화성

- HTML, PDF, ePub 변환을 고려해 비표준 Markdown 문법 사용을 줄인다.
- 변환 도구가 정해져 있다면 그 도구의 Markdown 확장 문법만 사용한다.
- 수식, 코드 하이라이트, 목차, 참고문헌은 변환 도구 지원 여부를 확인한다.
- Markdown 기준 원고는 출판 변환 전에도 독립적으로 읽을 수 있어야 한다.

## 9. MkDocs 파일명과 제목

- 챕터 파일명은 `.guide/OUTPUTS.md`의 `NN-<chapter-slug>.md` 규칙을 따른다.
- 챕터 파일명은 정렬과 URL을 위한 안정적인 영문 slug로 작성하고, 독자에게 표시할 제목은 문서의 첫 번째 `#` 제목으로 작성한다.
- 파일명 slug와 첫 번째 `#` 제목은 같은 주제를 가리켜야 한다.
- `index.md`는 사이트 또는 책의 진입점에만 사용한다.
- 모든 챕터에 `chapter.md` 같은 동일한 파일명을 반복해서 사용하지 않는다.
