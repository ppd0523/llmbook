# 기술자료 작성 에이전트 지침

이 파일은 에이전트가 기술 학습자료 작성을 시작할 때 가장 먼저 읽어야 하는 진입 지침이다.

에이전트는 이 파일을 읽은 뒤 `.guide/PROCESS.md`를 작업 프로토콜로 사용하고, `.guide/OUTPUTS.md`를 최종 산출물 계약으로 사용하며, `.guide/MARKDOWN_STYLE.md`를 Markdown 작성 기준으로 사용해야 한다.

## 1. 진입 규칙

기술자료 작성을 요청받으면 에이전트는 다음 순서로 진행한다.

1. `.guide/AGENTS.md`를 읽는다.
2. `.guide/PROCESS.md`를 읽고 작업 순서, 반복 조건, 완료 조건을 확인한다.
3. `.guide/OUTPUTS.md`를 읽고 최종 산출물 형식과 파일 구조 규칙을 확인한다.
4. `.guide/MARKDOWN_STYLE.md`를 읽고 Markdown 기준 원고 작성 규칙을 확인한다.
5. `.guide/01_scope.md`부터 `.guide/08_publish.md`까지 필요한 내부 작업 산출물 템플릿을 확인한다.
6. 사용자에게 자료 작성에 필요한 필수 정보와 최종 산출물 형식을 확인한다.
7. 필수 정보가 명확해진 뒤에만 `<자료폴더>/_work/01_scope.md` 단계부터 작업을 시작한다.

## 2. 기준 원고와 최종 산출물

- 기술자료의 기준 원고는 Markdown으로 작성한다.
- 최종 산출물은 사용자가 실제로 읽거나 배포할 수 있는 파일 또는 챕터 폴더 구조다.
- 최종 산출물 형식은 `.guide/OUTPUTS.md`를 따른다.
- 챕터형 Markdown의 기본 배치는 MkDocs 자동 탐색을 사용하는 `docs/<book-slug>/NN-<chapter-slug>.md` 구조로 한다.
- `01_scope.md`부터 `08_publish.md`까지의 단계 파일은 내부 작업 산출물이며 최종 학습자료가 아니다.
- 내부 작업 산출물은 기본적으로 `<자료폴더>/_work/` 아래에 둔다.
- PDF, HTML, ePub, PPT/PPTX 등은 Markdown 기준 원고에서 파생되는 출판 산출물로 취급한다.
- 출판 변환과 최종 파일 검수는 내용 작성 이후 `08_publish.md` 단계에서 별도로 다룬다.

## 3. 필수 확인 정보

에이전트는 초고를 작성하기 전에 다음 정보를 반드시 확보해야 한다.

### 3.1 주제

- 작성할 기술 주제는 무엇인가?
- 중심 질문은 무엇인가?
- 이 자료가 해결해야 하는 문제는 무엇인가?
- 다룰 기술 영역은 무엇인가?

### 3.2 대상 독자

- 독자의 숙련도는 초심자, 일부 지식이 있는 중급자, 실무 경험이 있는 중급자, 전문가 중 어디에 가까운가?
- 독자는 어떤 전공 또는 배경을 가지고 있는가?
- 독자가 이미 알고 있다고 가정할 수 있는 개념은 무엇인가?
- 독자가 모른다고 가정해야 하는 개념은 무엇인가?
- 독자는 구현, 실험, 디버깅, 논문 독해 경험이 있는가?

### 3.3 학습 목표

- 이 자료를 읽은 뒤 독자가 무엇을 할 수 있어야 하는가?
- 목표는 관찰 가능한 행동으로 표현되어 있는가?
- 개념 이해, 문제 풀이, 구현, 설계, 디버깅, 논문/기술문서 독해, 실무 적용 중 무엇을 우선하는가?

### 3.4 범위

- 반드시 포함해야 하는 내용은 무엇인가?
- 제외해야 하는 내용은 무엇인가?
- 심화 자료나 다음 장으로 넘겨야 하는 내용은 무엇인가?
- 수식, 코드, 그림, 표, 실습 중 무엇이 필요한가?

### 3.5 산출물 제약

- 원하는 분량은 어느 정도인가?
- 예제 또는 실습을 포함해야 하는가?
- 특정 언어, 도구, 프레임워크, 하드웨어, 논문, 표준을 사용해야 하는가?
- 참고문헌 또는 출처 수준에 대한 요구가 있는가?
- 최종 산출물 형식은 챕터형 Markdown, 단일 Markdown, 단일 PDF, 단일 PPT/PPTX 중 무엇인가?
- 최종 산출물 파일명 또는 MkDocs 책/챕터 경로 구조에 대한 요구가 있는가?
- Markdown 기준 원고 외에 PDF, HTML, ePub, PPT/PPTX 같은 변환 대상이 정해져 있는가?

## 4. 질문 규칙

에이전트는 필요한 정보가 부족하면 초고를 작성하지 말고 사용자에게 질문해야 한다.

- 한 번에 너무 많은 질문을 던지지 말고, 작성 진행을 막는 핵심 질문부터 묻는다.
- 사용자가 모호하게 답하면 다시 구체화해서 묻는다.
- 독자 상태, 학습 목표, 포함/제외 범위가 불명확하면 반드시 재질문한다.
- 최종 산출물 형식이 불명확하면 반드시 재질문한다.
- 사용자가 명시하지 않은 정보는 합리적으로 가정할 수 있지만, 중요한 가정은 `01_scope.md`에 기록한다.
- 기술 정확성에 영향을 주는 정보는 추측하지 않는다.
- 사용자에게서 충분한 정보가 확보되기 전에는 `04_draft.md` 초고를 작성하지 않는다.

## 5. 최소 시작 조건

다음 다섯 항목이 확인되어야 작업을 시작할 수 있다.

```text
주제:
대상 독자:
학습 목표:
분량/범위:
최종 산출물 형식:
```

이 다섯 항목 중 하나라도 비어 있으면 에이전트는 먼저 사용자에게 질문한다.

## 6. 권장 사용자 명령 형식

사용자는 다음 형식으로 기술자료 작성을 요청하는 것이 좋다.

```text
.guide/PROCESS.md를 작업 프로토콜로 따르고, .guide/MARKDOWN_STYLE.md를 Markdown 작성 기준으로 사용해라.
.guide/OUTPUTS.md를 최종 산출물 형식과 파일 구조 기준으로 사용해라.
내부 작업 산출물은 <자료폴더>/_work/ 아래에 두고, 최종 산출물은 사용자가 읽거나 배포할 수 있는 형태로 생성해라.

주제:
대상 독자:
학습 목표:
분량/범위:
최종 산출물 형식:
제약:
```

예:

```text
.guide/PROCESS.md를 작업 프로토콜로 사용해서 기술 학습자료를 작성해라.
.guide/MARKDOWN_STYLE.md를 Markdown 작성 기준으로 사용해라.
.guide/OUTPUTS.md에 따라 최종 산출물을 구성해라.

주제: PID 제어의 기본 원리와 튜닝 방법
대상 독자: 미적분과 기본 선형시스템은 알고 있지만 제어공학은 처음 배우는 학부 2~3학년
학습 목표: 계단 응답을 보고 P/I/D 항을 조정할 수 있게 한다.
분량/범위: 1장 분량, Worked Example 1개와 연습문제 3개 포함
최종 산출물 형식: 단일 Markdown 학습자료
제약: 수식은 필요하지만 직관적 설명을 먼저 제공한다.
```

## 7. 작업 시작 절차

사용자 요청이 들어오면 에이전트는 다음과 같이 행동한다.

1. 요청에서 `주제`, `대상 독자`, `학습 목표`, `분량/범위`, `최종 산출물 형식`을 추출한다.
2. 빠진 항목이 있으면 사용자에게 먼저 묻는다.
3. 사용자의 답변을 바탕으로 `<자료폴더>/_work/01_scope.md` 형식의 범위 정의를 작성한다.
4. `<자료폴더>/_work/01_scope.md`의 완료 조건을 만족하는지 확인한다.
5. 완료 조건을 만족하지 않으면 다시 질문한다.
6. 완료 조건을 만족하면 `.guide/PROCESS.md`의 순서에 따라 다음 단계로 진행한다.

## 8. 진행 중 되돌아가기 규칙

- 독자 상태가 흔들리면 `<자료폴더>/_work/01_scope.md`로 돌아간다.
- 출처나 기술 근거가 부족하면 `<자료폴더>/_work/02_research.md`로 돌아간다.
- 설명 순서가 불안정하면 `<자료폴더>/_work/03_outline.md`로 돌아간다.
- 본문 누락이 많으면 `<자료폴더>/_work/04_draft.md`로 돌아간다.
- 기술 오류가 발견되면 `<자료폴더>/_work/05_review.md`를 갱신하고 필요한 단계로 되돌아간다.
- 학습성 문제가 발견되면 `<자료폴더>/_work/06_revision.md`에서 수정 계획을 세운다.
- Markdown 규칙을 위반하면 `.guide/MARKDOWN_STYLE.md`에 맞게 수정한다.
- 출판 변환 문제는 `<자료폴더>/_work/08_publish.md`에서 처리하되, 내용 문제라면 이전 단계로 되돌린다.

## 9. 금지 사항

- 독자 상태를 확정하지 않고 초고를 작성하지 않는다.
- 최소 시작 조건이 비어 있는데도 임의로 작성하지 않는다.
- 출처 없는 정의, 수식, 알고리즘을 최종 Markdown 기준 원고나 최종 산출물에 단정적으로 넣지 않는다.
- 기술 검증 없이 최종 Markdown 기준 원고나 최종 산출물을 만들지 않는다.
- 여러 중심 질문을 한 장에 섞지 않는다.
- 초고 단계에서 문장 교정에 몰두하지 않는다.
- 최종 산출물에 내부 작업 메모, `TODO`, `검증 필요`, `출처 필요`를 남기지 않는다.
- 내부 작업 산출물만 만들고 학습자료 작성이 완료되었다고 판단하지 않는다.
- 번호가 붙은 내부 단계 파일을 자료 폴더 루트에 직접 두지 않는다.

## 10. 완료 기준

에이전트는 다음 조건을 만족할 때 기술자료 작성이 완료되었다고 판단한다.

- `<자료폴더>/_work/01_scope.md`부터 `<자료폴더>/_work/08_publish.md`까지 필요한 내부 작업 산출물이 작성되어 있다.
- 사용자가 선택한 최종 산출물 형식과 경로가 `<자료폴더>/_work/01_scope.md`에 기록되어 있다.
- 사용자가 읽거나 배포할 최종 산출물 파일 또는 챕터 폴더 구조가 실제로 생성되어 있다.
- 내부 작업 산출물이 자료 폴더 루트에 최종 산출물처럼 노출되어 있지 않다.
- 독자 상태, 학습 목표, 범위, 성공 기준이 명확하다.
- 핵심 기술 내용의 출처와 검증 상태가 기록되어 있다.
- 최종 Markdown 기준 원고가 학습 목표와 대응한다.
- 예제와 연습문제가 본문만으로 수행 가능하다.
- 최종 산출물에 내부 메모와 미검증 표시가 없다.
- 필요한 경우 `<자료폴더>/_work/08_publish.md`에 출판 변환 검수 결과가 기록되어 있다.
