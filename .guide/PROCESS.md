# 에이전트 작업 프로세스

이 문서는 기술 학습자료를 작성할 때 에이전트가 따라야 할 작업 순서, 반복 조건, 완료 조건을 정의한다.

산출물 템플릿은 내부 작업 파일에 무엇을 기록할지 정의하고, 이 문서는 어떻게 작업할지 정의한다. 사용자가 실제로 읽거나 배포할 최종 산출물의 형식과 파일 구조는 `.guide/OUTPUTS.md`를 따른다.

## 1. 기본 원칙

- 기준 원고는 Markdown이다.
- `07_final.md`는 최종 산출물이 아니라 최종 산출물을 만들기 위한 출판 전 기준 원고다.
- 사용자가 읽거나 배포할 최종 산출물은 `.guide/OUTPUTS.md`의 형식 중 하나로 생성한다.
- `01_scope.md`부터 `08_publish.md`까지의 단계 파일은 내부 작업 산출물이며 `<자료폴더>/_work/` 아래에 둔다.
- PDF, HTML, ePub, PPT/PPTX 등 출판 형식 변환은 `08_publish.md`에서 별도로 검수한다.
- 기술 검증, 구조 퇴고, 문장 교정은 같은 단계에서 처리하지 않는다.
- 모르는 내용을 추측하지 않는다.
- 정의, 수식, 알고리즘, 표준, 버전 의존 정보는 출처를 남긴다.
- 독자 상태가 불명확하면 초고를 작성하지 않는다.
- 최종 Markdown 기준 원고와 최종 산출물에는 `TODO`, `검증 필요`, `출처 필요`를 남기지 않는다.

## 2. 전체 흐름

```text
0. 독자 상태 진단
1. 작성 범위 정의
2. 자료 조사
3. 구성 설계
4. 초고 작성
5. 기술 검증
6. 구조 퇴고와 학습성 개선
7. 출판 전 최종 Markdown 원고 작성
8. 최종 산출물 구성과 출판 변환 검수
```

`0`부터 `7`까지는 기준 원고를 완성하기 위한 필수 단계다. `8`은 사용자가 선택한 최종 산출물 형식에 맞게 챕터 파일, 단일 Markdown, PDF, PPT/PPTX 등을 구성하고 검수하는 단계다.

## 2.1 단계 파일 경로 규칙

이 문서에서 `01_scope.md`부터 `08_publish.md`까지의 파일명은 내부 작업 산출물을 가리킨다. 실제 생성 위치는 다음과 같다.

```text
<자료폴더>/_work/01_scope.md
<자료폴더>/_work/02_research.md
<자료폴더>/_work/03_outline.md
<자료폴더>/_work/04_draft.md
<자료폴더>/_work/05_review.md
<자료폴더>/_work/06_revision.md
<자료폴더>/_work/07_final.md
<자료폴더>/_work/08_publish.md
```

자료 폴더 루트에 번호가 붙은 내부 단계 파일을 직접 두지 않는다.

## 3. 단계별 절차

### 0. 독자 상태 진단

목적: 자료를 읽을 독자의 현재 상태를 확정한다.

입력:

- 사용자 요청
- 기존 문서 또는 강의 계획
- 예상 독자 정보

작업:

- 독자의 숙련도를 분류한다.
- 선행지식을 확정한다.
- 학습 목적을 확인한다.
- 독자가 막힐 가능성이 높은 지점을 기록한다.

출력:

- `<자료폴더>/_work/01_scope.md`의 `독자 상태 진단` 섹션

완료 조건:

- 숙련도, 선행지식, 학습 목적이 명시되어 있다.
- 독자가 모른다고 가정할 개념이 분리되어 있다.
- 자료에서 새로 설명할 개념이 분리되어 있다.

다음 단계로 넘어가면 안 되는 경우:

- 독자 수준이 "일반 독자"처럼 모호하다.
- 선행지식이 비어 있다.
- 학습 목적이 개념 이해, 구현, 설계, 디버깅 중 어디에 가까운지 알 수 없다.

### 1. 작성 범위 정의

목적: 자료의 중심 질문, 포함 범위, 제외 범위, 출력 형식을 확정한다.

입력:

- 독자 상태 진단 결과
- 사용자 요구사항

작업:

- 중심 질문을 하나로 정리한다.
- 학습 목표를 행동 중심으로 작성한다.
- 포함할 내용과 제외할 내용을 분리한다.
- 기준 원고 형식을 Markdown으로 명시한다.
- `.guide/OUTPUTS.md`에 따라 최종 산출물 형식과 경로를 기록한다.
- 필요한 경우 향후 변환할 출판 형식과 도구를 기록한다.

출력:

- `<자료폴더>/_work/01_scope.md`

완료 조건:

- 중심 질문이 하나다.
- 학습 목표가 관찰 가능한 행동으로 작성되어 있다.
- 포함 범위와 제외 범위가 분리되어 있다.
- 성공 기준과 최종 과제의 방향이 보인다.
- 기준 원고 형식이 Markdown으로 명시되어 있다.
- 최종 산출물 형식과 경로가 명시되어 있다.

되돌아가기 조건:

- 작성 중 새로운 독자층이 발견되면 독자 상태 진단으로 돌아간다.
- 자료가 여러 중심 질문을 다루기 시작하면 범위를 다시 나눈다.

### 2. 자료 조사

목적: 정확한 정의, 수식, 알고리즘, 표준, 구현 근거를 확보한다.

입력:

- `<자료폴더>/_work/01_scope.md`

작업:

- 교과서, 논문, 공식 문서, 표준 문서, 구현체를 우선 조사한다.
- 블로그, 강의, 비공식 문서는 보조 출처로만 사용한다.
- 정의, 수식, 알고리즘, 버전 의존 정보의 출처를 기록한다.
- 논쟁점이나 설명 방식 차이를 기록한다.

출력:

- `<자료폴더>/_work/02_research.md`

완료 조건:

- 핵심 정의와 주요 기술 주장의 출처가 있다.
- 출처가 없는 항목은 `출처 필요`로 표시되어 있다.
- 버전 또는 조건에 의존하는 내용이 표시되어 있다.
- 예제 후보와 위험 요소가 기록되어 있다.

되돌아가기 조건:

- 조사 결과 주제가 너무 넓으면 `01_scope.md`로 돌아가 범위를 줄인다.
- 핵심 정의를 검증할 수 없으면 초고로 넘어가지 않는다.

### 3. 구성 설계

목적: 학습자가 따라갈 수 있는 설명 순서를 만든다.

입력:

- `<자료폴더>/_work/01_scope.md`
- `<자료폴더>/_work/02_research.md`

작업:

- 문제 제기에서 요약까지의 흐름을 설계한다.
- 개념 의존성을 정리한다.
- 예제와 연습문제를 학습 목표에 대응시킨다.
- 위험 구간과 오해 가능 지점을 표시한다.

출력:

- `<자료폴더>/_work/03_outline.md`

완료 조건:

- 섹션 순서가 개념 의존성과 맞는다.
- 새 용어가 정의 없이 먼저 등장하지 않는다.
- 각 핵심 개념에 예제 또는 확인 질문이 대응된다.
- 최종 연습문제가 학습 목표와 대응된다.

되돌아가기 조건:

- 설명 순서가 독자의 선행지식과 맞지 않으면 구성을 다시 설계한다.
- 핵심 예제를 만들 수 없으면 조사 단계로 돌아간다.

### 4. 초고 작성

목적: 전체 내용을 읽을 수 있는 형태로 만든다.

입력:

- `<자료폴더>/_work/03_outline.md`

작업:

- 전체 섹션을 채운다.
- 문장 완성도보다 설명 흐름과 누락 방지를 우선한다.
- 불확실한 내용은 `TODO`, `검증 필요`, `출처 필요`로 표시한다.
- 예제, 연습문제, 흔한 오류를 초안 수준으로 포함한다.

출력:

- `<자료폴더>/_work/04_draft.md`

완료 조건:

- 처음부터 끝까지 읽을 수 있다.
- 주요 개념, 예제, 연습문제, 요약이 모두 존재한다.
- 불확실한 내용이 표시되어 있다.
- 기술 검증할 대상이 충분히 드러나 있다.

금지 사항:

- 초고 단계에서 문장 교정에 과도한 시간을 쓰지 않는다.
- 검증하지 않은 기술 내용을 단정하지 않는다.

### 5. 기술 검증

목적: 초고의 기술적 정확성을 확인한다.

입력:

- `<자료폴더>/_work/04_draft.md`
- `<자료폴더>/_work/02_research.md`

작업:

- 정의, 수식, 단위, 기호, 알고리즘, 코드, 그림, 표를 검증한다.
- 코드 예제는 가능하면 실행한다.
- 수식과 알고리즘은 가정과 적용 조건을 확인한다.
- 출처와 본문 주장의 대응을 확인한다.

출력:

- `<자료폴더>/_work/05_review.md`

완료 조건:

- 주요 정의와 기술 주장의 검증 상태가 기록되어 있다.
- 코드 또는 계산 예제의 검증 결과가 기록되어 있다.
- 미검증 항목과 남은 위험이 분리되어 있다.

되돌아가기 조건:

- 기술 오류가 핵심 설명에 영향을 주면 `04_draft.md`로 돌아간다.
- 출처가 부족해 검증할 수 없으면 `02_research.md`로 돌아간다.
- 주제 범위 자체가 잘못되었으면 `01_scope.md`로 돌아간다.

### 6. 구조 퇴고와 학습성 개선

목적: 설명 순서, 학습성, 예제, 연습문제를 개선한다.

입력:

- `<자료폴더>/_work/04_draft.md`
- `<자료폴더>/_work/05_review.md`

작업:

- 기술 검증 결과를 반영한다.
- 설명 순서를 독자 관점에서 재배치한다.
- 추상 개념에 구체 예제를 붙인다.
- Worked Example과 직접 풀이 문제를 함께 배치한다.
- 흔한 오류와 실패 사례를 보강한다.

출력:

- `<자료폴더>/_work/06_revision.md`
- 수정된 본문

완료 조건:

- 학습 목표와 본문이 대응한다.
- 연습문제가 본문 내용만으로 풀린다.
- 추상 설명과 구체 사례가 연결된다.
- 기술 검증에서 나온 필수 수정이 반영되어 있다.

되돌아가기 조건:

- 구조 문제가 크면 `03_outline.md`로 돌아간다.
- 누락된 개념이 많으면 `04_draft.md`로 돌아간다.
- 기술 검증 결과와 충돌하면 `05_review.md`를 갱신한다.

### 7. 출판 전 최종 Markdown 원고 작성

목적: 사용자가 선택한 최종 산출물을 만들기 전에 내용이 확정된 Markdown 기준 원고를 만든다.

입력:

- 퇴고 반영본
- `<자료폴더>/_work/06_revision.md`
- `MARKDOWN_STYLE.md`

작업:

- 문장, 용어, 표기법을 정리한다.
- Markdown 구조를 정리한다.
- 내부 메모와 TODO를 제거한다.
- 참고문헌과 추가 읽을거리를 정리한다.
- `.guide/OUTPUTS.md`의 최종 산출물 형식으로 변환하거나 분리할 수 있는 구조인지 확인한다.

출력:

- `<자료폴더>/_work/07_final.md`

완료 조건:

- 최종 Markdown 기준 원고에 내부 작업 메모가 없다.
- `TODO`, `검증 필요`, `출처 필요`가 없다.
- 제목 계층, 코드 블록, 수식, 표, 링크가 Markdown 규칙을 따른다.
- 학습 목표, 본문, 예제, 연습문제가 서로 대응한다.
- 최종 산출물 형식에 맞게 챕터 분리 또는 변환이 가능한 상태다.

되돌아가기 조건:

- 최종 교정 중 기술 오류가 발견되면 `05_review.md`로 돌아간다.
- 학습 목표와 본문이 맞지 않으면 `06_revision.md`로 돌아간다.

### 8. 최종 산출물 구성과 출판 변환 검수

목적: Markdown 기준 원고를 사용자가 선택한 최종 산출물 형식으로 구성하고, 필요한 경우 출판 변환 결과를 검수한다.

입력:

- `<자료폴더>/_work/07_final.md`
- `<자료폴더>/_work/01_scope.md`
- `.guide/OUTPUTS.md`
- 변환 도구 설정 또는 챕터 분리 기준

작업:

- 챕터형 Markdown, 단일 Markdown, PDF, PPT/PPTX 등 사용자가 선택한 최종 산출물을 생성한다.
- HTML, PDF, ePub, PPT/PPTX 등 필요한 형식으로 변환한다.
- 변환 결과에서 렌더링, 링크, 수식, 표, 코드, 그림을 확인한다.
- 자료 폴더 루트에서 최종 산출물이 바로 식별되는지 확인한다.
- 내부 작업 산출물이 최종 산출물처럼 노출되지 않았는지 확인한다.
- 형식 문제는 내용 수정과 분리해서 기록한다.

출력:

- 사용자가 선택한 최종 산출물 파일 또는 챕터 폴더 구조
- `<자료폴더>/_work/08_publish.md`

완료 조건:

- 최종 산출물 형식과 결과 파일 또는 챕터 구조가 기록되어 있다.
- 변환 명령과 결과 파일이 기록되어 있다.
- 형식별 검수 항목이 확인되어 있다.
- 변환 과정에서 발견된 내용 문제는 앞 단계로 되돌려 처리한다.

## 4. 반복 규칙

- 독자 상태가 흔들리면 `01_scope.md`부터 다시 확인한다.
- 출처나 기술 근거가 부족하면 `02_research.md`로 돌아간다.
- 설명 순서가 불안정하면 `03_outline.md`로 돌아간다.
- 본문 누락이 많으면 `04_draft.md`로 돌아간다.
- 기술 오류가 발견되면 `05_review.md`를 갱신하고 필요한 단계로 되돌아간다.
- 학습성 문제가 발견되면 `06_revision.md`에서 수정 계획을 세운다.
- 최종 Markdown 기준 원고가 깨지면 `MARKDOWN_STYLE.md`에 맞게 `07_final.md`를 고친다.
- 최종 산출물 구성 또는 출판 변환 문제는 `08_publish.md`에서 처리하되, 내용 문제라면 이전 단계로 되돌린다.

## 5. 에이전트 판단 규칙

- 독자에게 필요한 최소 선행지식을 먼저 확인한다.
- 한 장에는 중심 질문을 하나만 둔다.
- 중요한 개념은 정의, 직관, 예제, 적용 조건을 함께 제공한다.
- 수식은 기호, 단위, 가정, 적용 범위를 함께 설명한다.
- 코드 예제는 가능한 한 실행 검증한다.
- 회로, 프로토콜, 로봇 시스템, 네트워크 구조처럼 상태와 흐름이 중요한 내용은 그림 또는 표를 계획한다.
- 논쟁적이거나 버전 의존적인 내용은 조건을 명시한다.
- 불확실한 내용은 최종 산출물에 넣기 전에 검증한다.
- Markdown 기준 원고의 가독성을 우선하고, PDF/HTML/PPT 조판 문제는 출판 단계에서 다룬다.

## 6. 금지 사항

- 독자 상태를 확정하지 않고 초고를 작성하지 않는다.
- 출처 없는 정의, 수식, 알고리즘을 최종 Markdown 기준 원고나 최종 산출물에 단정적으로 넣지 않는다.
- 기술 검증 없이 최종 Markdown 기준 원고나 최종 산출물을 만들지 않는다.
- 여러 중심 질문을 한 장에 섞지 않는다.
- 초고 단계에서 문장 교정에 몰두하지 않는다.
- 출판 형식 문제 때문에 내용 구조를 성급하게 바꾸지 않는다.
- 최종 산출물에 내부 작업 메모, TODO, 검증 필요 표시를 남기지 않는다.
- 내부 작업 산출물만 만들고 작업을 완료했다고 판단하지 않는다.
- 번호가 붙은 내부 단계 파일을 자료 폴더 루트에 직접 두지 않는다.

## 7. 최종 완료 조건

기술자료 작성 작업은 다음 조건을 만족할 때 완료된다.

- `<자료폴더>/_work/01_scope.md`부터 `<자료폴더>/_work/08_publish.md`까지 필요한 내부 작업 산출물이 작성되어 있다.
- `<자료폴더>/_work/01_scope.md`에 최종 산출물 형식과 경로가 기록되어 있다.
- 사용자가 선택한 최종 산출물 파일 또는 MkDocs 책/챕터 파일 구조가 실제로 생성되어 있다.
- 내부 작업 산출물이 자료 폴더 루트에 최종 산출물처럼 노출되어 있지 않다.
- 독자 상태, 학습 목표, 범위, 성공 기준이 명확하다.
- 핵심 기술 내용의 출처와 검증 상태가 기록되어 있다.
- 최종 Markdown 기준 원고가 학습 목표와 대응한다.
- 예제와 연습문제가 본문만으로 수행 가능하다.
- 최종 산출물에 내부 메모와 미검증 표시가 없다.
- 필요한 경우 `<자료폴더>/_work/08_publish.md`에 출판 변환 검수 결과가 기록되어 있다.
